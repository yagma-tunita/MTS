================================================================================
        航运物流管理系统 - 后端开发指南 (Oracle数据库)
        版本: 1.0
        目标: 基于提供的 tables.sql 数据库模式
================================================================================

1. 数据库连接配置
--------------------------------------------------------------------------------
- 使用连接池 (HikariCP, Druid) 连接 Oracle 数据库。
- JDBC URL 示例:
  jdbc:oracle:thin:@//host:1521/service_name
- 驱动类: oracle.jdbc.OracleDriver
- 建议设置连接超时、空闲回收、测试查询 "SELECT 1 FROM DUAL"。

2. 实体类映射 (ORM)
--------------------------------------------------------------------------------
- 表名与字段名均使用小写 + 下划线 (如 shipping_order)。
- 实体类字段使用驼峰命名 (如 shippingOrder)，通过 ORM 映射 (MyBatis 或 JPA 注解)。
- 所有表均有 create_time, update_time (部分有 delete_time)，实体类中对应为 Date/LocalDateTime 类型。

3. 主键生成策略
--------------------------------------------------------------------------------
- 数据库已为每张表创建 SEQUENCE (seq_表名) 和 BEFORE INSERT 触发器。
- 触发器会自动为主键列赋值 (若传入 NULL)。
- 后端代码中，插入新记录时**不要**手动设置主键值 (设为 null)，由触发器自动填充。
- 若需要返回生成的主键：
   * MyBatis: 使用 <selectKey> 或 useGeneratedKeys="true" keyProperty="id" (注意 Oracle 需要特殊配置)
   * 推荐做法: 插入后立即查询 seq_xxx.currval (但注意会话隔离问题)，或使用 RETURNING INTO 子句。
   * 更简单: 插入前后端调用序列 nextval (而非依赖触发器)，但需保持全局一致性。当前设计触发器已足够。

4. 软删除 (Soft Delete) 处理
--------------------------------------------------------------------------------
- 所有业务表 (city, shipper_company, shipping_company, port, berth, vessel, shipping_line, shipping_order, order_cargo) 均有 delete_time 字段。
- 删除操作: 执行 UPDATE 设置 delete_time = SYSDATE，不要物理 DELETE。
- 查询时: 必须在 WHERE 条件中加上 "AND delete_time IS NULL"，过滤掉已删除记录。
- 建议在 DAO/Repository 层编写基础查询方法时，自动附加 delete_time IS NULL 条件。
- 唯一索引 (如 order_no, imo_number) 如果包含软删除记录，无法重复插入已删除的相同唯一值。业务上允许重复需调整唯一索引为条件唯一 (Oracle 不支持部分索引，可用函数索引 + CASE)，或后端先清理 delete_time 再插入。推荐保留唯一性约束，禁止重复 (即使已删除也视为冲突)。

5. 时间字段处理 (create_time, update_time)
--------------------------------------------------------------------------------
- 数据库无自动更新 update_time 的触发器，必须由后端代码在每次更新时手动设置 update_time = 当前时间。
- 插入时: 必须设置 create_time = 当前时间，update_time 可为 null 或同时设置为当前时间。
- 统一使用服务器时区 (建议 UTC 或东八区)，数据库字段 DATE 类型包含年月日时分秒。
- Java 推荐使用 java.time.LocalDateTime，JDBC 驱动会自动转换。

6. 枚举与状态字段
--------------------------------------------------------------------------------
- 以下字段建议在后端定义枚举类，数据库存储数字或短字符串：
   * admin.role (NUMBER): 1-超级管理员, 2-普通管理员 (默认2)
   * shipping_order.payment_status (NUMBER): 0-未支付,1-已支付,2-部分支付等
   * shipping_order.order_status (NUMBER): 0-草稿,1-已确认,2-运输中,3-已完成,4-已取消等
   * account_status (shipper_company/shipping_company): 1-启用,0-禁用,2-锁定
   * berth.is_available, vessel.is_available: 1-可用,0-不可用
   * voyage_cargo_note.operation_type (VARCHAR): 'LOAD' 装货, 'DISCHARGE' 卸货
- 校验: 后端接口接收枚举时，校验合法性后转数值存储；查询返回枚举。

7. 外键关联加载策略
--------------------------------------------------------------------------------
- 表之间存在外键关系，但数据库未定义级联操作 (ON DELETE ...)。后端需自行管理：
   * 删除父表记录 (软删除) 前，应检查是否存在子表引用 (若业务不允许孤儿数据)，可查询子表是否有 delete_time IS NULL 且外键指向该父记录。
   * 例如：删除 vessel 前，检查 voyage_berthing 和 voyage_cargo_note 中是否还有未删除的记录。
- 查询关联数据时，推荐使用 JOIN 并带上子表的 delete_time IS NULL 条件，避免返回已软删除的关联对象。

8. 索引使用建议
--------------------------------------------------------------------------------
- 脚本已为所有外键列创建索引，无需额外操心。
- 高频查询场景建议补充复合索引 (后端开发者可根据慢查询日志提出需求)：
   * voyage_cargo_note 按 (line_id, voyage_date) 查询累计货运量
   * voyage_berthing 按 (port_id, actual_arrival_time) 查询港口拥堵情况
   * shipping_order 按 (shipper_company_id, order_status) 查询订单列表
- 注意: 软删除列 delete_time 已有索引，但复合索引中包含 delete_time 效果更好 (如 (status, delete_time))。

9. 事务管理
--------------------------------------------------------------------------------
- 涉及多表操作 (如创建订单并写入 order_cargo、segment_capacity_usage) 必须使用 @Transactional 确保原子性。
- 事务隔离级别: 默认 READ COMMITTED 即可。
- 处理并发冲突: 对于 capacity 占用 (segment_capacity_usage)，建议使用乐观锁 (增加 version 字段) 或悲观锁 (SELECT FOR UPDATE)。

10. 大字段处理 (CLOB)
--------------------------------------------------------------------------------
- shipping_line 的 port_sequence 和 description 为 CLOB 类型。
- 在 MyBatis 中映射为 String，注意 JDBC 驱动默认限制 4000 字符，需配置 orai18n.jar 或设置连接属性 SetBigStringTryClob=true。
- 插入或更新时，使用 #{clobField, jdbcType=CLOB} 方式。

11. 订单号生成策略
--------------------------------------------------------------------------------
- shipping_order.order_no 要求唯一，后端需实现生成唯一订单号算法：
   * 推荐: 前缀 + 日期(YYYYMMDD) + 序列号(如 Redis 自增或数据库序列)。
   * 即使有唯一索引，生成时最好先检查是否存在，避免插入失败。

12. 运力管理逻辑 (关键业务)
--------------------------------------------------------------------------------
- voyage_cargo_note 中的 cumulative_booked_capacity_ton 字段表示累计已预订吨位，后端在插入新 cargo_note 时需计算：
   * 根据同一 (line_id, vessel_id, voyage_date) 之前的记录累加 weight_ton 得到累计值。
   * 注意并发更新: 使用 SELECT FOR UPDATE 锁定该航次的所有记录，或者使用乐观锁。
- segment_capacity_usage 记录每个订单占用的分段运力，需要在创建订单时插入，取消订单时删除或标记。

13. 密码存储
--------------------------------------------------------------------------------
- shipper_company.login_password, shipping_company.login_password, admin.password 字段长度 VARCHAR2(255)。
- 后端必须使用 bcrypt 或 PBKDF2 加密存储，绝对禁止明文。
- 登录接口验证时，比对加密后的值。

14. 分页查询
--------------------------------------------------------------------------------
- Oracle 12c+ 推荐使用 OFFSET FETCH 语法:
  SELECT * FROM shipping_order WHERE delete_time IS NULL
  ORDER BY create_time DESC
  OFFSET :offset ROWS FETCH NEXT :pageSize ROWS ONLY;
- 低版本可使用 ROWNUM 子查询。

15. 常见错误与规避
--------------------------------------------------------------------------------
- 触发器赋主键 + MyBatis useGeneratedKeys 可能无法正确返回主键，建议改用 <selectKey> 查询序列。
- 日期比较时注意数据库 DATE 包含时间部分，若仅需日期可用 TRUNC(create_time) = TO_DATE(...)。
- 连接池中的连接若长时间空闲，Oracle 会断开，需配置 keepalive 或验证查询。
- 序列缓存 (CACHE 100) 在数据库重启后可能跳过一段号，若订单号不允许跳号，可改为 NOCACHE (但性能下降)。

16. 测试数据准备
--------------------------------------------------------------------------------
- 开发/测试环境建议编写初始化脚本，插入至少：
   * 一条 admin 记录 (用户名 admin, 加密密码)
   * 几个城市、港口、泊位
   * 一家航运公司和一条船舶
   * 一条航运线路和航次数据
- 方便联调。

17. 日志与监控
--------------------------------------------------------------------------------
- 记录每个关键 SQL 的执行时间 (慢查询阈值 >200ms)。
- 跟踪订单状态变更日志，便于排查。

================================================================================
                            开发原则总结
================================================================================
1. 后端负责: 业务枚举校验、update_time 维护、软删除过滤、事务边界、密码加密。
2. 数据库负责: 数据持久化、唯一约束、主键自增、外键完整性（非级联）、性能索引。
3. 沟通约定: 任何表结构变更需同步更新此文档；复合索引需求由后端性能分析提出。
